# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

FinPlan — a goal-based investment planner AI agent (LangChain v1 on LangGraph). A user states a goal in natural language ("₹2L now, ₹15k/month, 12 years, goal ₹50L, moderate risk") and the agent produces a risk profile, asset allocation, projection, Monte-Carlo range, feasibility analysis, and inflation-adjusted target. Indian-market focus (yfinance, NSE tickers, ₹).

## Commands

All commands run from the repo root with the venv Python (`.venv\Scripts\python.exe` on Windows).

```bash
# Setup
python -m venv .venv
.venv\Scripts\python.exe -m pip install -r requirements.txt
# Add an API key to .env (copy .env.example); GROQ_API_KEY is the working provider

# Tests (offline, no API key, ~3s)
.venv\Scripts\python.exe -m pytest -q
.venv\Scripts\python.exe -m pytest tests/test_planner.py -q        # one file
.venv\Scripts\python.exe -m pytest tests/test_planner.py::test_generate_plan_chains_values -q  # one test

# The live agent test is opt-in (makes real LLM calls); enable with an env var:
#   PowerShell:  $env:RUN_LIVE_AGENT_TEST='1'; .venv\Scripts\python.exe -m pytest tests/test_agent_smoke.py -q

# Run
.venv\Scripts\python.exe -m streamlit run app.py     # web UI (needs API key)
.venv\Scripts\python.exe -m uvicorn finplan.api.main:app --reload  # REST API + OpenAPI docs at /docs (/plan needs NO key)
.venv\Scripts\python.exe -m scripts.chat             # terminal chat (needs API key)
.venv\Scripts\python.exe -m scripts.demo_engine      # deterministic engine demo (NO API key)

# Agent evaluation harness — runs the live agent on a golden set (needs API key)
.venv\Scripts\python.exe -m finplan.eval             # add --judge for LLM-as-judge scoring
```

## The core architectural principle

**The LLM never does arithmetic, unit conversion, or number formatting.** Every financial figure is computed/parsed/formatted in deterministic Python; the LLM only orchestrates tools, interprets results, and copies pre-formatted strings. This is the project's central design decision — preserve it in any change. Concretely:

- **Computation** lives in `finplan/tools/*.py` (pure functions, no LLM, fully unit-tested with known answers).
- **`finplan/parsing.py`** does input/output number handling: `parse_amount("50 lakh") -> 5_000_000` and `format_inr(4851170) -> "₹48,51,170"`. The LLM passes amounts verbatim (e.g. `"50 lakh"`) and copies pre-formatted display strings — it is *not* trusted to convert lakh/crore or group digits (smaller models drop/add digits).

## Layered architecture (read these to understand the big picture)

1. **`finplan/tools/`** — the deterministic engine. Pure functions:
   - `market_data.py` — yfinance fetch; `compute_asset_stats` filters corrupt ticks (>50% daily moves) so a bad print can't poison volatility. `DEFAULT_TICKERS` maps equity/debt/gold to NSE symbols.
   - `projection.py` — `future_value` / `project_growth` (lump sum + SIP, monthly compounding). `future_value` is the single source of truth reused by the feasibility solvers.
   - `risk.py` — risk profiling (`min(capacity, willingness)` rule), allocation glide-path, portfolio blending. Accepts synonyms ("moderate"→"medium").
   - `simulation.py` — `monte_carlo_simulation` (seeded; p10/median/p90 + goal probability).
   - `feasibility.py` — solvers that invert `future_value` to find required SIP / years / return.

2. **`finplan/planner.py`** — `generate_plan()`: the **composite tool** that chains the whole pipeline in Python. This exists for two reasons: (a) it guarantees each step consumes the previous step's exact output (the LLM can't inject guessed intermediate values, which it does when it calls granular tools in parallel), and (b) it collapses a plan from ~8 LLM calls to ~2. It returns a `summary` dict of pre-formatted display strings the LLM is instructed to quote verbatim. `_class_stats` fetches real per-class data with a graceful fallback to assumed returns if yfinance fails.

3. **`finplan/agent/`** — the LLM layer.
   - `tools.py` — thin `@tool` wrappers over the engine. `generate_plan` is primary; granular tools are for "what-if" follow-ups. Money params are typed `float | str` and run through `parse_amount`. `ALL_TOOLS` is the registry.
   - `prompts.py` — `SYSTEM_PROMPT`; enforces "use generate_plan once, never compute/convert/format numbers yourself, copy from summary".
   - `planner_agent.py` — `build_agent()` (LangChain v1 `create_agent` + `InMemorySaver` checkpointer for conversation memory) and `run_turn()`. `content_to_text()` flattens provider-specific content blocks (Gemini/Anthropic return a list, not a str).

4. **`finplan/memory/store.py`** — SQLite persistence for saved plans + progress log (survives restarts; complements the in-process conversation checkpointer).

5. **`app.py`** — Streamlit UI: chat + charts auto-generated by parsing the agent's tool outputs (`generate_plan`'s nested result, with fallback to granular tool outputs) + a tool-call trace expander.

6. **`finplan/api/`** — a FastAPI service over the *same* engine/agent (thin wrappers, no logic duplicated). `main.py` exposes `GET /healthz`, `POST /plan` and `POST /plan/multi` (**deterministic — no LLM key needed**; the blocking engine runs via `asyncio.to_thread`), and `POST /chat` (agent turn; returns 503 without a key). `schemas.py` holds the Pydantic models whose money fields accept text like `"50 lakh"` through a `parse_amount` validator. Auto OpenAPI docs at `/docs`.

7. **`finplan/observability.py`** — a LangChain `MetricsCallbackHandler` capturing per-turn token usage, latency, and LLM/tool-call counts; `/chat` returns them in a `metrics` block.

8. **`finplan/eval/`** — the agent evaluation harness: a golden set (`dataset.py`) + three deterministic checks (tool selection, input extraction, verbatim numeric grounding) in `runner.py`, plus an optional LLM-as-judge. Run via `python -m finplan.eval`; the pure scoring functions are unit-tested offline (`tests/test_eval.py`).

## Model configuration (`finplan/config.py`)

All models are created via `get_chat_model(role)` and selected from `.env` as `provider:model_id` (or a preset alias in `PRESETS`). Swapping providers is a one-line `.env` change — no code edits. `ORCHESTRATOR_MODEL` drives the agent loop; `ADVISOR_MODEL` is reserved for a final-synthesis model. Reasoning models (matched by `_REASONING_HINTS`) are sent without a `temperature` arg.

**Provider note:** Groq (`groq:llama-3.3-70b-versatile`) is the working default — it has a generous free tier. The Gemini free tier on the current key is effectively unusable (very low/zero RPD). Llama is fast and cheap but weak at Indian-number conversion and formatting, which is exactly why `parse_amount` / `format_inr` / the `summary` strings exist — keep that scaffolding when changing models or tools.

## Serving, tooling & CI

- **Docker** — multi-stage `Dockerfile` (slim, non-root) serving the API via uvicorn; `docker-compose.yml` for local runs; `render.yaml` is a one-click Render deploy blueprint. The image is built and smoke-tested in CI.
- **CI** — `.github/workflows/ci.yml` runs `ruff` + `mypy` (scoped to `finplan/api`, `observability.py`, `eval`, `parsing.py`) + `pytest` on every push/PR to `main`, then a `docker build`. No secrets needed — the live agent test is opt-in, so the suite is deterministic.
- **`pyproject.toml`** centralizes ruff/mypy/pytest config. Note `[tool.pytest.ini_options] pythonpath = ["."]` — required so `import finplan` resolves under a bare `pytest` (how CI invokes it), not only `python -m pytest`.

## Gotchas

- LangChain v1 is installed: use `langchain.agents.create_agent` (LangGraph-based), **not** the legacy `AgentExecutor` / `create_tool_calling_agent`.
- `load_dotenv()` resolves `.env` relative to the calling file's directory. Code inside the `finplan` package finds it; scripts run from elsewhere should use `load_dotenv(find_dotenv(usecwd=True))`.
- When adding a tool that takes money, type the param `float | str` and run it through `parse_amount`; return any user-facing figure via `format_inr` (and add it to a `summary` if it goes through `generate_plan`).
